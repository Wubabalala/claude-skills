# Documentation Convention

> **The single source of truth for the multi-project documentation system.**
>
> - `project-onboarding` **generates** structures per this convention (Phase 1 output).
> - `doc-garden` **audits** structures against this convention (`check_doc_convention()` in normalize mode).
>
> Born from two real production projects: VectorDraft (Java standalone) and V-Story (Spring Cloud microservice). See §5 for live examples.

---

## §1. Naming Convention

### Root triplet (required)

Every project root must have **three** files with distinct responsibilities. Do not collapse them into one:

| File | Audience | Responsibility |
|---|---|---|
| `CLAUDE.md` | AI coding agents | Architecture traps, decisions, large-file map, operational constraints, server runbook hooks |
| `AGENTS.md` | AI entry navigation | The first file an AI reads. Lists required reading order: `OVERVIEW.md` → `CLAUDE.md` → business `*-ref.md` / module entry points. **Template in §4.** |
| `README.md` | Human users | Project intro + quickstart. Never overwritten by onboarding — it belongs to humans. |

### `docs/` required files by doc system level

| File | Purpose |
|---|---|
| `docs/OVERVIEW.md` | One-page project map. Generated by onboarding Phase 1. Required at `simple` and `standard` levels. |
| `docs/architecture-traps.md` | Trap log. Every debugging session that took > 30 min lands a dated entry here. Format: `## YYYY-MM-DD 现象` then root cause + fix. Append-only. Required at `standard` level. |
| `docs/TRUTH_SOURCES.md` | Human-readable index of "who owns which fact." Required at `standard` level. |
| `docs/QUICK_REFERENCE.md` | Volatile fact table for line counts, large files, ports, and fast lookup. Required at `standard` level. |

### Doc system levels

`doc_system_level` describes documentation depth. It is independent from
`project_type`, which describes deployment topology.

| Level | When created | Required structure |
|---|---|---|
| `simple` | Phase 0/1 onboarding | Root triplet + `docs/OVERVIEW.md` |
| `standard` | Phase 2b governance upgrade | `simple` + traps + truth sources + quick reference + references + plans/ops/archive |

`simple` is the minimum entry layer for every project. `standard` is the
governance layer reached after deeper project understanding.

### Time-anchored files

- `docs/plans/{YYYY-MM-DD}-{kebab-slug}.md` — in-flight design plans. Date prefix lets them sort chronologically and ages out via grep.
- When a plan completes, move it to `docs/archive/plans/` (keep the date prefix).

### Memory files

Memory file format (frontmatter + structure) is defined in `memory-file-spec.md` — not duplicated here.

## §2. Placement Convention

| Path | Contents |
|---|---|
| `docs/plans/` | In-flight plans (dated). Move to `docs/archive/plans/` on completion. |
| `docs/ops/` | Operations runbooks: deploy, backup, rollback, monitoring queries, log query recipes. |
| `docs/references/` | Code anchor deep-references and domain truth source files. One pair per business domain when needed: `<domain>-ref.md` for code anchors and `<domain>-truth-source.md` for authoritative invariants. |
| `docs/archive/` | Historical archive. Anything that's no longer current but worth keeping for context. |

Anything outside this taxonomy goes flat under `docs/` only if it's a top-level cross-cutting topic (e.g., `service-topology.md`, `auth-architecture.md` in microservice projects).

## §2.1 Truth Source Format

`docs/TRUTH_SOURCES.md` is the index. It points to owners; it does not
duplicate all invariants.

```markdown
# Truth Sources

## <Domain Name>

- Truth source: `docs/references/<domain>-truth-source.md`
- Scope: <facts covered by this source>
- Owner: <file/module/system that owns the fact>
- Override policy: `must-not-duplicate | may-reference | append-only`
- Review rule: <must/never rule extracted by code-review>
```

`docs/references/<domain>-truth-source.md` owns the domain invariants:

```markdown
# <Domain> Truth Source

## Scope
<what this file owns>

## Invariants
- **TRUTH**: <claim> — source: `<path>`
- **TRUTH**: <claim> — source: `<path>`

## Out Of Scope
<what this file does not own>
```

## §3. Layering Convention — three project types

The skeleton depends on `project_type`. This is aligned with `doc-garden`'s `SKELETONS` dict (three types: `standalone`, `monorepo`, `microservice`).

| Type | Detection criteria | Skeleton |
|---|---|---|
| `standalone` | Single deployable, single build chain (one `pom.xml` / `package.json` for the whole repo, or clear single-app layout). | **Root triplet + root `docs/` only.** No per-module `CLAUDE.md`. **Example: VectorDraft.** |
| `monorepo` | Multiple modules in one repo, but **unified build / single deployable** (e.g., frontend + backend built and released together). | **Default: root-only** (root triplet + root `docs/`; submodules may have only `README.md`). If a project has already configured `layer2` module docs (via `.claude/doc-garden.json`), `doc-garden` will not reject them — it continues auditing per the existing `SKELETONS["monorepo"]["module"]` skeleton. |
| `microservice` | Multiple **independently deployable** services (each builds and deploys on its own). | Root triplet + root `docs/` (cross-service topics: `service-topology.md`, `auth-architecture.md`). **Each service subdirectory replicates the full four-piece entry kit: `CLAUDE.md` + `AGENTS.md` + `README.md` + `docs/`.** **Example: V-Story.** |

### `microservice` detection heuristics (onboarding-time)

Use these as **signals**, not absolute rules. If any are hit, default to `microservice` and confirm with the user in Phase 0.4:

- Multiple independent manifest files (multiple `pom.xml` / `package.json`), each carrying its own `build` / `package` script.
- Top-level service orchestration markers: Spring Cloud Gateway / Nacos / Eureka / Consul, `docker-compose.yml` with multiple `services:`, k8s manifests with multiple `Deployment` resources.
- Multiple independent `Dockerfile` files at sibling subdirectories.

If none hit → default to `monorepo` or `standalone` per the monorepo glob check; let the user override in Phase 0.4.

## §4. AGENTS.md template

This is the canonical template. `project-onboarding` Phase 1 generates AGENTS.md by filling in the slots below. **SKILL.md does not duplicate this template** — it references this section.

### Common shell (all project types)

```markdown
# {ProjectName} — 仓库入口导航

> **默认简体中文协作。** 这是 Claude / AI 工具进入本仓库的第一站。

---

## 必读（按顺序）

1. [`docs/OVERVIEW.md`](docs/OVERVIEW.md) — 项目地图、业务域、文档导航
2. [`CLAUDE.md`](CLAUDE.md) — 架构陷阱、决策、大文件、运维约束
3. 业务域 `docs/references/*.md`（开始具体任务前读对应那份）：
   - {domain-1}: [`{ref-1}.md`](docs/references/{ref-1}.md)
   - {domain-2}: [`{ref-2}.md`](docs/references/{ref-2}.md)

---
```

If the detected primary doc language is English, swap the leading note to: `> **English by default.** This is the first file Claude / AI tools should read when entering this repo.`

### Additional section for `microservice` only

```markdown
## 模块入口

- {module-A}：[`{module-A}/AGENTS.md`](./{module-A}/AGENTS.md)
- {module-B}：[`{module-B}/AGENTS.md`](./{module-B}/AGENTS.md)
- ...
```

### Per-service AGENTS.md (microservice only)

Each service subdirectory's `AGENTS.md` follows the same shell but:
- Points back to root: `> Root entry: [`../AGENTS.md`](../AGENTS.md)`
- Lists only this service's own `docs/` files
- Omits the cross-service "模块入口" section
