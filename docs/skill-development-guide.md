# Skill 开发与优化指南

从 project-onboarding skill 的实际开发过程中提炼的方法论。
适用于创建新 skill 和优化已有 skill。

---

## 创建新 Skill

### 第 1 步：明确定位

回答三个问题：
- **谁用**：目标用户画像（新手？资深？特定角色？）
- **何时触发**：用户会说什么话来触发这个 skill
- **输出什么**：skill 执行完后产出的具体产物

### 第 2 步：设计工作流

将 skill 的执行过程拆分为**有确认门的阶段**：

```
阶段 N: 做什么 (读/写属性)
    ↓ 展示结果 → 用户确认
阶段 N+1: ...
```

设计原则：
- 每个阶段结束后等用户确认，用户可随时停止
- 读操作和写操作分开，写操作前必须有确认
- 批量工具调用优先（减少轮次，降低 token 消耗）

### 第 3 步：编写 SKILL.md

**Frontmatter 要求：**
- `description` 用第三人称："This skill should be used when..."
- 包含具体触发短语（用户实际会说的话）
- 描述控制在 500 字符以内

**Body 要求：**
- 全文用祈使语气（"Scan the files"，不是 "You should scan"）
- 主体控制在 1,500-2,500 词
- 每条声明引用来源文件
- 模板是上限不是下限——没数据的段落直接跳过

### 第 4 步：渐进式披露

```
skill-name/
├── SKILL.md           ← 核心流程（1,500-2,500 词）
├── references/        ← 详细规则，按需加载
│   ├── xxx-rules.md
│   └── xxx-spec.md
├── examples/          ← 可复制的示例（可选）
└── scripts/           ← 工具脚本（可选）
```

提取标准：
- 安全规则、格式规范等**独立于流程**的内容 → `references/`
- 在 SKILL.md 中用一句话指向："See `references/xxx.md` for full rules"
- **核心工作流逻辑留在 SKILL.md**，不要拆出去

### 第 5 步：实测验证

在真实项目上跑一遍完整流程，记录：
- 哪些步骤执行顺畅
- 哪些步骤产出质量差（说明 skill 指引不够明确）
- token 消耗是否合理
- 用户交互轮次是否过多

## 优化已有 Skill

### 第 1 步：评审

用 `plugin-dev:skill-reviewer` agent 做结构化评审，关注：

| 维度 | 关注点 |
|------|--------|
| 描述质量 | 第三人称？触发短语覆盖全？ |
| 内容质量 | 祈使语气？有来源引用？ |
| 渐进式披露 | 主体词数？该拆的拆了没？ |
| 安全性 | 写入前有检查？用户确认门？ |

### 第 2 步：竞品对比

```bash
npx skills find "你的skill关键词"
```

拉取同类 skill 的源码，建立对比矩阵：
- 功能覆盖差异
- 你有别人没有的（差异化卖点）
- 别人有你没有的（潜在改进方向）

### 第 3 步：针对性改进

**优先级排序原则：改动小、收益大、与现有流程兼容的先做。**

常见改进方向：
- **渐进式披露**：主体超 3,000 词 → 提取 reference 文件
- **流程补充**：实测中发现的缺失步骤
- **降低用户负担**：AI 能推断的不要问用户（Infer before Ask）
- **视觉输出**：文字描述改为图表（Mermaid 等）

### 第 4 步：实测 + 评分

在真实项目上重新跑一遍，对比改进前后：
- 功能是否正常
- 产出质量是否提升
- token 消耗是否降低

## 评分体系

| 维度 | 满分标准 |
|------|---------|
| 描述触发覆盖 /10 | 第三人称，多触发短语，有隐式触发指导 |
| 工作流设计 /10 | 阶段清晰，有确认门，可随时停止 |
| 安全性 /10 | 写入前阻断检查，用户逐项确认 |
| 写作质量 /10 | 祈使语气一致，表格信息密度高，无废话 |
| 渐进式披露 /10 | 主体 ≤2,500 词，详细内容在 references/ |
| 实用性 /10 | 输出可直接使用，不需二次加工 |
| 竞品差异化 /10 | 有独特功能，零竞品覆盖 |

综合分 = 各维度加权平均。8.5+ 为高质量 skill。

## 关键教训（从 project-onboarding 实践中总结）

1. **Infer before Ask** — AI 先从代码推断 why，标注置信度，只把低置信问题留给用户。否则用户会觉得 skill 在偷懒。

2. **新人场景不能丢** — 如果用户答不上来（真正的新人），AI 的推断应该写入产出并标记 `[inferred — to be verified]`，而不是直接跳过。

3. **渐进式披露拆的是参考资料，不是核心流程** — 安全规则、文件格式规范可以拆；工作流步骤、确认门逻辑必须留在主体里。

4. **视觉输出价值高** — 一张 Mermaid 架构图比 500 字描述更快建立心智模型。如果 skill 产出文档，优先考虑加图。

5. **Token 预算意识** — 每次 skill 触发都会加载主体到上下文。3,000 词和 1,500 词的差距在长对话中会累积。
